# Инструкции для агента: работа с обновлениями версий структуры

Этот документ описывает, как агенту действовать при работе над новой версией структуры и при релизе. План работ — в корне репозитория: `_Structure-Implementation-Plan.md`.

---

## Роли папок

| Папка | Назначение |
|-------|------------|
| **`_Structure/`** | Текущая версия инструкций. Единый источник правды для агента в обычных задачах. **Не** меняется при разработке новой версии. |
| **`_Structure-Management/`** | Разработка и история: версии (v0.1, v0.2, …), отчёты (Reports/), черновики (_Drafts/). В `.cursorignore` — агент по умолчанию её не видит. |

---

## Когда пользователь работает над структурой

1. **Проверить видимость _Structure-Management**
   - Если пользователь просит править инструкции в `_Structure-Management/`, создать v0.X, добавить отчёт в Reports и т.п. — напомнить: для доступа к папке нужно временно убрать из `.cursorignore` строку `**/_Structure-Management/*` (или пользователь открывает нужные файлы вручную).
   - В начале сессии уточнить: «Сейчас работаем в _Structure-Management; _Structure не трогаем — это текущая версия для всех».

2. **Создание новой версии (например v0.5)**
   - Создать папку `_Structure-Management/v0.5/` и подпапку `_Structure-Management/v0.5/_Job_Descriptions/`.
   - Скопировать в v0.5 содержимое предыдущей версии (v0.4): все файлы из корня v0.4 и из `_Job_Descriptions/`.
   - Дальнейшие правки вносить только в файлы внутри `v0.5/`.

3. **Правки в версии**
   - Редактировать только файлы в указанной версии (например `v0.5/`). Не менять `_Structure/` и не перезаписывать другие версии.
   - Крупные черновики можно готовить в `_Structure-Management/_Drafts/`, затем переносить в соответствующую версию.

4. **Отчёты по структуре**
   - Новые отчёты (аудит, план, changelog) размещать в `_Structure-Management/Reports/ГГГГ-ММ-ДД/`, по аналогии с существующими датами. Имена файлов — с датой и кратким назначением (например `2026-02-15-Audit-Plan.md`).

---

## Релиз новой версии (копирование в _Structure)

Выполнять **только после явного указания пользователя** (например: «выкати v0.5 в _Structure», «релиз v0.5»).

1. **Скопировать содержимое версии в _Structure**
   - Из `_Structure-Management/v0.X/` (например v0.5) скопировать в `_Structure/`:
     - файлы из корня версии: `Structure_Instruction.md`, `Rules_Instruction.md`, `Organizational_Roles.md`, `Project_Deployment_Template.md`, `Interaction_Instruction.md`, `Human_Behavior_Guide.md`, `Structure_Analysis.md`;
     - содержимое `_Job_Descriptions/` (все `JD_*.md`, `README.md`).
   - Файлы в `_Structure/` **заменить** содержимым из выбранной версии (не смешивать с текущим состоянием).

2. **Обновить README в _Structure-Management**
   - В `_Structure-Management/README.md` обновить блок «История изменений» и указать: текущая версия в `_Structure/` — v0.X (дата релиза).

3. **Не менять при релизе**
   - `_Structure/README.md` и `_Structure/AGENT_Structure_Version_Workflow.md` не перезаписывать из версии; при необходимости править их отдельно (краткие пояснения и инструкции для агента).

---

## Обычный режим (не работа над структурой)

- Опираться только на **`_Structure/`**: `Structure_Instruction`, `Rules_Instruction`, `Organizational_Roles`, JD и т.д.
- Не предлагать правки в `_Structure-Management/` и не копировать оттуда файлы в _Structure без явного запроса пользователя на работу над структурой или релиз.
