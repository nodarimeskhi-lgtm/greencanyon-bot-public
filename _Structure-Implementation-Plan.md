**Источник:** `_Structure-Management/v0.4/` (актуальная версия на 2026-02-09).

**Статус:** план выполнен 2026-02-09. Инструкции для агента по работе с обновлениями версий — в `_Structure/AGENT_Structure_Version_Workflow.md`.

## Шаг 3. Добавить README.md в _Structure

Создать `_Structure/README.md` с кратким пояснением:

- Это **текущая** версия инструкций по структуре (один источник правды для агента).
- Версии и отчёты по разработке структуры — в `_Structure-Management/`.
- При релизе новой версии (например v0.5) обновить содержимое `_Structure/` из соответствующей папки в `_Structure-Management/`.

Добавить в README блок:

- При утверждении новой версии — скопировать её содержимое в `_Structure/`.

## Релиз следующей версии (v0.5 и далее)

1. Разработать новую версию в `_Structure-Management/` (например `v0.5/`).
2. После утверждения: скопировать содержимое `v0.5/` в `_Structure/`, заменив текущие файлы.
3. В `_Structure-Management/README.md` обновить «История изменений» и указать актуальную версию в `_Structure/`.

## Работа с агентом в _Structure-Management (после внедрения изменений)

В обычном режиме агент **не видит** `_Structure-Management/` (папка в `.cursorignore`). Чтобы вместе с агентом править инструкции, версии или отчёты по структуре, нужно временно «включить» эту папку.

### Вариант A: Временно убрать из .cursorignore (удобно для сессии правок)

1. **Перед началом работы над структурой**
   - Открыть `.cursorignore`.
   - Закомментировать или удалить строку: `**/_Structure-Management/*`.
   - Сохранить файл.

2. **Во время сессии**
   - Агент видит `_Structure-Management/`: может искать по файлам, читать версии, править черновики в `_Drafts/`, создавать `v0.5/` и т.д.
   - В промпте можно явно указать: «работаем в _Structure-Management», «редактируй Structure_Instruction в v0.5», «добавь отчёт в Reports/2026-02-XX».

3. **После окончания сессии**
   - Вернуть в `.cursorignore` строку: `**/_Structure-Management/*`.
   - Сохранить. Обычная работа снова идёт только с `_Structure/`.

### Вариант B: Работа через открытые файлы (без изменения .cursorignore)

- Открыть в редакторе нужные файлы из `_Structure-Management/` (например из `v0.5/` или `_Drafts/`).
- В чате с агентом явно указать: «редактируй открытый файл _Structure-Management/v0.5/Structure_Instruction.md» или приложить файл через @.
- Агент может править только те файлы, которые вы ему явно дали; поиск по всей папке у него по-прежнему недоступен.

**Когда что использовать:** вариант A — при большой правке (новая версия, несколько файлов, отчёт). Вариант B — при точечном изменении одного-двух файлов без желания трогать `.cursorignore`.

### Типовой цикл разработки новой версии (v0.5) с агентом

1. **Включить видимость** _Structure-Management (вариант A выше).

2. **Создать ветку версии**
   - Попросить агента: «Создай в _Structure-Management папку v0.5 и скопируй туда содержимое v0.4 как основу».
   - Или скопировать `v0.4/` в `v0.5/` вручную, затем попросить агента вносить правки в `v0.5/`.

3. **Правки в v0.5**
   - Формулировать задачи по файлам/темам: «В v0.5 в Structure_Instruction добавь раздел про …», «В Rules_Instruction измени правило про …», «В _Job_Descriptions обнови JD_CTO с учётом …».
   - Черновики крупных изменений можно готовить в `_Structure-Management/_Drafts/`, затем переносить в `v0.5/`.

4. **Отчёты по работе над структурой**
   - Новые отчёты (аудит, план, changelog) класть в `_Structure-Management/Reports/ГГГГ-ММ-ДД/` по аналогии с существующими.
   - Агент может помочь сформировать отчёт, если папка видна (вариант A) или если вы открыли нужный файл/шаблон.

5. **Релиз v0.5**
   - После утверждения: скопировать содержимое `_Structure-Management/v0.5/` в `_Structure/` (вручную или попросить агента при включённой видимости _Structure-Management).
   - Обновить `_Structure-Management/README.md`: история изменений, указание «текущая версия в _Structure — v0.5».

6. **Выключить видимость** _Structure-Management (вернуть строку в `.cursorignore`).

### Что держать в голове

- **Контекст для агента:** при работе «в _Structure-Management» в первом сообщении сессии полезно написать: «Сейчас работаем над структурой: папка _Structure-Management включена в контекст / открыты файлы из v0.5. Не трогай _Structure — это текущая версия для всех».
- **Единый источник правды в обычной работе:** в повседневных задачах (не по разработке структуры) агент опирается только на `_Structure/`; `_Structure-Management/` в обычном режиме не используется.
- **Документирование:** в `_Structure-Management/README.md` или в плане (этом файле) можно кратко описать выбранный способ работы (A или B) и цикл релиза, чтобы не забыть при следующей итерации.
